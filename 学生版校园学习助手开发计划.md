
# 学生版校园学习助手开发计划（工作日每天2小时，周末不限）

## 目标定义
- 项目定位：陪伴式学习助手（朋友感聊天 + 资料整理 + 可扩展多模态）
- 技术路线：本地服务网关 + 远程大模型API + 本地记忆与检索
- 产出标准：可在线演示、可复现部署、可量化效果、可写进简历

## 时间预算
- 工作日：每天2小时
- 周末：建议每天4到8小时
- 总周期：4周（可扩展到6周）

## 每周里程碑

### 第1周：最小可用聊天助手（MVP-1）
目标：让助手稳定聊天，支持人设和会话连续。

工作日任务（每晚2小时）
- D1：梳理项目结构，跑通后端启动、健康检查、聊天接口
- D2：补齐远程API环境变量配置，验证本地到远程的请求链路
- D3：增加系统人设注入（朋友感、鼓励式、简洁表达）
- D4：增加会话ID机制，按会话保存最近N轮历史
- D5：增加基础日志（请求耗时、是否成功、错误原因）

周末任务
- 完成端到端联调：连续对话10轮无崩溃
- 增加异常兜底回复（超时、接口失败时给用户可理解提示）
- 产出周报：完成点、问题、下周计划

验收标准
- 可稳定进行多轮聊天
- 健康检查可用
- 错误有可读日志

### 第2周：学习资料整理与检索（MVP-2）
目标：支持资料上传、切片、检索、带引用回答。

工作日任务（每晚2小时）
- D1：设计资料数据结构（课程、文档、片段、标签）
- D2：接入文档解析（先做txt和pdf）
- D3：实现切片与入库（SQLite + 本地向量库）
- D4：实现检索接口（关键词 + 向量相似度）
- D5：把检索结果拼接到聊天上下文

周末任务
- 实现回答引用来源（文档名 + 段落）
- 增加资料管理能力（列表、删除、按课程过滤）
- 构建一个小测试集（20个问题）做命中率测试

验收标准
- 上传资料后可问答
- 回答含引用来源
- 20个问题中至少70%可命中有效内容

### 第3周：朋友感与长期记忆（MVP-3）
目标：让助手有连续人格和学习陪伴感。

工作日任务（每晚2小时）
- D1：拆分记忆层：短期记忆、长期偏好、学习进度
- D2：实现偏好提取（作息、课程、目标）并持久化
- D3：实现记忆检索注入（回答前取最相关记忆）
- D4：实现主动提醒策略（DDL、复习计划）
- D5：实现学习卡片生成（知识点卡片、错题卡片）//再说

周末任务
- 调整语气策略（鼓励、共情、不过度说教）
- 做一次真实体验走查（连续使用2小时）
- 修复高频问题（答非所问、重复、冗长）

验收标准
- 用户偏好可持续生效
- 助手可根据历史给个性化建议
- 回复风格稳定

### 第4周：多模态可扩展底座 + 简历包装
目标：先打好可扩展架构，并完成展示材料。

工作日任务（每晚2小时）
- D1：抽象模型路由层（文本、图像、语音统一接口）
- D2：预留图像输入接口（先返回占位结果）
- D3：预留语音接口（先接入STT或TTS其中一个）
- D4：增加成本统计（请求次数、估算token、估算费用）
- D5：补充配置文档和快速启动脚本

- D6：前端UI多模态与美感优化设计，提升视觉体验和交互流畅度
- D7：设计并实现前端设置页面，支持参数（如模型、记忆、风格等）动态调整
- D8：后端联网RAG功能设计与实现，支持外部知识检索增强生成（如联网搜索、API检索等）

周末任务
- 完成演示版页面或演示脚本
- 录制2到3分钟演示视频
- 完成简历项目描述（STAR写法）

验收标准
- 架构支持后续接入图像和语音模型
- 有演示视频、仓库文档、运行说明
- 能讲清楚性能、成本、可扩展性



### 第5周：类人长时记忆机制（MVP-4）
目标：让助手具备自然、连贯的长时记忆能力，支持历史事件的智能回忆。

前端配合任务：
- 优化前端多模态展示与交互体验，支持文本、图片、语音等多模态内容自然切换
- 完善设置页面，支持记忆参数、风格切换等后端能力的前端可视化配置

工作日任务（每晚2小时）
- D1：调研并设计分层记忆架构（短期/长期/回忆触发）
- D2：实现历史对话embedding存储与向量检索（Chroma/FAISS/SQLite+embedding）
- D3：开发回忆触发机制（关键词/实体识别+相似度检索）
- D4：集成记忆检索与上下文动态拼接逻辑
- D5：测试多种回忆场景，优化召回相关性与上下文长度控制

周末任务
- 增加记忆摘要与遗忘机制（定期摘要、清理无用记忆）
- 体验走查：模拟跨天/跨话题连续对话，验证回忆自然度
- 完善开发文档，记录记忆机制设计与使用说明

验收标准
- 助手能在用户提及历史事件时，自动检索并补充相关记忆
- 回忆内容相关性高，未主动提及时不会乱插历史
- 记忆库可控，支持定期清理与摘要
- 有详细的记忆机制设计文档

#### 意图检测与风格切换补充

- 集成意图检测模型（如BERT/ERNIE/Roberta微调、fasttext、HuggingFace意图分类API等），支持区分“正常聊天”与“需要帮助”意图。
- “正常聊天”意图下，回复风格拟人、轻松、可适当开玩笑，无需敬语，回复简洁。
- “需要帮助”意图下，回复风格认真、聚焦问题本身，可省略敬语但不敷衍。
- 根据意图自动切换回复风格，提升对话自然度和实用性。
- 相关任务细化：
  - D1：调研并集成意图检测模型，支持意图分类
  - D2：实现风格切换逻辑，适配不同意图
  - D3：测试不同意图下的对话体验，优化风格切换
#### 主动回忆与高效检索补充

- 主动回忆触发机制：每轮对话自动进行意图/主题检测，判断是否与历史事件、偏好、知识点相关，无需用户显式提醒即可主动检索相关记忆。
- embedding相似度结合关键词/实体识别，自动筛选高相关历史，设定高相关性阈值，仅在高度相关时插入回忆，避免无关干扰。
- embedding检索采用本地向量库（如Chroma/FAISS），保证毫秒级响应，对历史记忆分层摘要和索引，优先检索近期和高频事件。
- 支持异步/多线程预检索，提升回忆速度。
- 可定期对历史对话做主题聚类，形成记忆标签，进一步提升检索效率。


## 每天2小时的推荐节奏
- 前10分钟：回顾昨天进度与今天目标
- 中间90分钟：只做一个核心任务
- 最后20分钟：记录日志、提交代码、写下明日待办

## 技术栈建议（学生友好）
- 后端：FastAPI
- 数据：SQLite（起步）+ 可选Redis
- 检索：向量库任选一个轻量方案
- 模型：远程API为主，本地模型为备选
- 前端：先命令行或简页，后续再做完整UI

## 面试常问与准备
- 为什么用远程模型而不是本地模型
- 你如何控制成本和延迟
- RAG为什么能提升准确率
- 记忆如何防止污染和过拟合用户偏好
- 如果并发上来你怎么扩展

## 风险与应对
- 风险：API费用超预算
  - 应对：设置日预算阈值，超限降级到轻量模型
- 风险：回答不稳定
  - 应对：固定系统提示词 + 引用约束 + 评测集回归
- 风险：开发时间碎片化
  - 应对：任务拆到90分钟可完成的粒度
